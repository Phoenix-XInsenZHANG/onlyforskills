# Bug Fix to Ship

从 Bug 报告到生产环境修复的完整工作流，确保修复质量可追溯、可验证。

## 触发条件

- 用户说"修复bug"、"修一下这个bug"
- 用户说"生产环境有问题"、"线上故障"
- 用户说"紧急修复"、"hotfix"
- 用户报告错误信息、异常行为

## 工作流程

```
┌─────────────────┐
│   1. DEBUGGING  │  分析根因，定位问题
└────────┬────────┘
         ▼
┌─────────────────┐
│      2. TDD     │  先写失败测试，再写修复
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. VERIFICATION │  验证修复有效，所有测试通过
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. CODE-REVIEW  │  请求代码审查
└────────┬────────┘
         ▼
┌─────────────────┐
│5. FINISH-BRANCH │  合并/PR/部署
└─────────────────┘
```

## 执行步骤

### Step 1: 调用 debugging skill

```
使用 Skill 工具调用 "debugging"
```

**目的**: 系统性分析 Bug 根因

**产出**:
- 根因分析报告
- 问题定位（文件、行号、函数）
- 复现步骤

### Step 2: 调用 tdd skill

```
使用 Skill 工具调用 "tdd"
```

**目的**: 测试驱动修复，先写失败测试再写修复代码

**产出**:
- 失败的测试用例（复现 bug）
- 修复代码
- 测试通过

### Step 3: 调用 verification skill

```
使用 Skill 工具调用 "verification"
```

**目的**: 验证修复有效，确保没有引入新问题

**产出**:
- 所有测试通过
- 验证命令输出
- 修复确认

### Step 4: 调用 requesting-code-review skill

```
使用 Skill 工具调用 "requesting-code-review"
```

**目的**: 请求代码审查，确保修复质量

**产出**:
- PR 或审查请求
- 审查反馈

### Step 5: 调用 finishing-branch skill

```
使用 Skill 工具调用 "finishing-branch"
```

**目的**: 完成分支工作，合并到主分支或部署

**产出**:
- 合并/PR/部署决策
- 完成的修复

## 重要规则

1. **顺序执行**: 必须按 1→2→3→4→5 顺序调用，不可跳过
2. **自动流转**: 每个 skill 完成后立即调用下一个，无需用户确认
3. **跳过交互**: 跳过 skill 内部的交互确认步骤，使用默认值继续
4. **完整执行**: 必须跑完全部 5 个 skill 才算完成
5. **证据优先**: 在声称修复完成前，必须有测试通过的证据

## 错误处理

如果某个 skill 失败：
1. 报告错误详情
2. 等待用户指示是否继续或修复
3. 不跳过任何步骤

## 示例

**用户输入**:
> 生产环境有一个严重bug：用户提交订单后偶尔出现重复扣款。订单表叫sc_order，支付通过Directus hook触发外部支付网关。

**Claude 行为**:
1. 调用 `debugging` → 分析 Directus hook 和支付网关调用时序，发现并发请求导致重复触发
2. 调用 `tdd` → 编写复现重复扣款的测试，实现幂等性修复
3. 调用 `verification` → 运行所有测试，验证修复有效
4. 调用 `requesting-code-review` → 创建 PR，请求审查
5. 调用 `finishing-branch` → 合并到 main，准备部署

## 与其他 Skill 的关系

| Skill | 关系 |
|-------|------|
| `full-feature-lifecycle` | 新功能开发用那个，bug 修复用这个 |
| `receiving-code-review` | 如果审查有反馈，用这个处理 |
| `e2e-test` | 修复后可调用进行端到端测试 |
